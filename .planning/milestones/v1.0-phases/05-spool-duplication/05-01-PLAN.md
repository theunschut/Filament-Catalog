**Edit 1 — Add openDuplicateDialog function.**

Place the new function immediately after openEditDialog (around line 302). The function signature and body:

```javascript
function openDuplicateDialog(spool) {
    resetFormForAdd();
    repopulateOwnerSelect(allOwners);
    // Pre-fill all fields from source spool (mirrors populateFormForEdit but stays in add mode)
    nameInput.value     = spool.name;
    matInput.value      = spool.material;
    const hex = HEX_RE.test(spool.colorHex ?? '') ? spool.colorHex : '#888888';
    colorPicker.value   = hex;
    colorHexInput.value = hex;
    colorSwatch.style.background = hex;
    ownerSelect.value   = String(spool.ownerId);
    weightInput.value   = spool.weightGrams ?? '';
    priceInput.value    = spool.pricePaid  ?? '';
    statusSelect.value  = spool.spoolStatus;
    paymentSelect.value = spool.paymentStatus;
    notesInput.value    = spool.notes ?? '';
    // Duplicate mode: title changes, stays in add mode (no editId, no delete button)
    dialogTitle.textContent  = 'Duplicate Spool';
    dialog.dataset.editId    = '';
    deleteBtn.style.display  = 'none';
    deleteConfirm.style.display = 'none';
    dialog.showModal();
}
```

Notes on the logic:
- `resetFormForAdd()` is called first — it already clears form, hides delete btn, and sets `dialog.dataset.editId = ''`. The subsequent assignments override specific fields.
- `dialogTitle.textContent = 'Duplicate Spool'` OVERRIDES the `'Add Spool'` title set by `resetFormForAdd()` — place it AFTER the reset call.
- `dialog.dataset.editId = ''` is set explicitly after field population as a defensive guard (resetFormForAdd sets it too, but this makes the intent clear).
- `deleteBtn.style.display = 'none'` and `deleteConfirm.style.display = 'none'` are also set defensively after reset (reset already hides them, but explicit is correct).
- All user-supplied string values set via `.value` assignment (form inputs) — not innerHTML. XSS safe.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| spool data -> DOM | Spool field values from API response rendered into form inputs |
| form inputs -> POST /api/spools | User-edited form values submitted as new spool payload |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | XSS (Injection) | openDuplicateDialog — field population | mitigate | All spool fields written to form inputs via `.value` assignment (not innerHTML). `dialogTitle.textContent` used for title. `notesInput.value` for notes. No raw HTML insertion of user data anywhere in the new code. |
| T-05-02 | Spoofing | POST /api/spools — duplicate save path | accept | Duplicate save follows the identical code path as Add (saveBtn handler checks `dialog.dataset.editId === ''` → calls `createSpool(payload)`). No new trust boundary; existing validation applies. |
| T-05-03 | Tampering | Original spool modified during duplicate | accept | `openDuplicateDialog` does not call any update API. The source `spool` object is only read for field values. `dialog.dataset.editId = ''` ensures saveBtn handler routes to `createSpool` not `updateSpool`. |
| T-05-04 | Information Disclosure | Spool data exposed in dialog pre-fill | accept | This is a local-only app (localhost:5000, Windows service). Spool data is already visible in the row the user clicked. Pre-filling the form exposes no new data. |
</threat_model>
