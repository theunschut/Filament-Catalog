## v1 Requirements

### Catalog Sync

- [x] **SYNC-01**: User can trigger a Bambu catalog sync via a "Sync Bambu catalog" button in the UI — *Completed in Phase 3*
- [x] **SYNC-02**: Sync reads product and color variant data from the locally installed Bambu Studio filament profile (`filaments_color_codes.json`) — no web requests required — *Completed in Phase 3*
- [x] **SYNC-03**: Color hex values are read directly from the `fila_color` field in `filaments_color_codes.json`; alpha channel is stripped from 8-digit hex values via `NormalizeHex()` — *Completed in Phase 3*
- [x] **SYNC-04**: Sync upserts into the BambuProduct table, matching on Name + Material to avoid duplicates; updates `LastSyncedAt` on each record — *Completed in Phase 3*
- [x] **SYNC-05**: UI shows when the catalog was last synced; displays a progress indicator while sync is running (202 + polling `/api/sync/status`) — *Completed in Phase 3*
- [x] **SYNC-06**: After the first sync the app works fully offline — BambuProduct table is the source of truth — *Completed in Phase 3*

### Spool Management

- [x] **SPOOL-01**: User can add a spool by selecting a product from a searchable catalog dropdown (color swatch preview shown on selection) — *Completed in Phase 2 + Phase 3 (catalog integration)*
- [x] **SPOOL-02**: User can assign a spool to an owner (self or a named friend) — *Completed in Phase 2*
- [x] **SPOOL-03**: User can set weight (grams), price paid, payment status (paid / unpaid / partial), spool status (sealed / active / empty), and optional notes — *Completed in Phase 2*
- [x] **SPOOL-04**: User can edit any spool's fields after creation — *Completed in Phase 2*
- [x] **SPOOL-05**: User can delete a spool — *Completed in Phase 2*
- [x] **SPOOL-06**: User can filter and search the spool list by owner, material type, spool status, payment status, and free text — *Completed in Phase 2*

### Owner Management

- [x] **OWNER-01**: User can add a named owner — *Completed in Phase 2*
- [x] **OWNER-02**: User can delete an owner — rejected with an error if the owner still has spools — *Completed in Phase 2*
- [x] **OWNER-03**: On first run (empty DB), app seeds one owner with `IsMe = true` and name "Me" — *Completed in Phase 1*

### Summary & Balance

- [x] **BAL-01**: Summary bar at top of page shows: total spools, my spools, total value, total owed to me — *Completed in Phase 2*
- [x] **BAL-02**: Balance overview section shows one row per non-me owner: name, spool count, total value, amount owed (sum of unpaid/partial spool prices) — *Completed in Phase 2, moved to sidebar in Phase 6*
- [x] **BAL-03**: Balance row visually flags when one or more contributing spools have no price set (so the total looks incomplete) — *Completed in Phase 2*

### Infrastructure

- [x] **INFRA-01**: App runs as a Windows service (auto-starts with Windows, accessible at `http://localhost:5000`) — *Completed in Phase 1*
- [x] **INFRA-02**: SQLite database file (`filament.db`) is stored relative to the executable using `AppContext.BaseDirectory` — not a relative working-directory path — *Completed in Phase 1*
- [x] **INFRA-03**: Single-page UI (`index.html`) served as static files from ASP.NET Core (`UseDefaultFiles` + `UseStaticFiles`) — *Completed in Phase 1*

## v2 Requirements (deferred to next milestone)

- Weight remaining on spool (grams) — status is sufficient for v1
- Partial payment amount tracking (running ledger) — partial treated as unpaid in v1
- Clickable balance rows filtering spool list to that owner
- Multi-region Bambu store support (non-EU)
- Export to CSV

---

## Out of Scope

- Authentication / login — local use only, single user
- Mobile / cross-platform — Windows service is sufficient
- AMS slot tracking — out of scope for inventory management
- QR / NFC labels — unnecessary complexity for v1
- AngleSharp HTML scraping — replaced by local Bambu Studio file
- Shopify JSON API / ImageSharp swatch extraction — replaced by local `filaments_color_codes.json`
